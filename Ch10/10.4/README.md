This problem has a 'breaking' change with the addition of go.mods. Because this repo bundles every chapter under one go.mod (gobook), the "whole workspace" go.list ... sweeps in dependencies from unrelated chapters (e.g., golang.org/x/crypto, pulled in transitively by some other chapter's use of x/net) that have nothing to do with whatever target package you're actually asking about. In the book's original per-project-GOPATH world, your workspace would only contain packages relevant to what you were working on, so this noise wouldn't occur.
